## C. PWA 구축 (W5~10, 6주) — 본 데모를 어떻게 발전시킬 것인가

### C-1. 본 Streamlit 데모에서 그대로 가져갈 것
| 자산 | PWA 이식 방식 |
|---|---|
| `core/prompts.py` | 그대로 사용 (FastAPI 서버에 동일 시스템 프롬프트) |
| `core/llm.py` | 서버 사이드 Claude API 호출 로직 그대로 |
| `core/rag.py` | Supabase pgvector 마이그레이션 (스키마: id, title, date, summary, embedding) |
| `core/badge.py` | 클라이언트 사이드 JS로 포팅 (badge 색상·아이콘 동일) |
| `data/sample_sillok.json` | DB 시드 데이터로 활용 + 공공데이터 청크 통합 |

### C-2. PWA에서 추가될 기능
- 지도 (카카오맵 SDK)
- GPS Geofence
- 카메라 Vision API (Claude Vision)
- 결제 (토스 + Eximbay)
- 음성 출력 (브라우저 내장 TTS, 핵심 장면만 ElevenLabs)
- Service Worker (오프라인 폴백)
- 만 14세 미만 보호자 동의 흐름

### C-3. 추천 기술 스택 (사업계획서 §2 기술 아키텍처와 일치)
- 프론트: Next.js 또는 React 14 + Tailwind + PWA 매니페스트
- 백엔드: FastAPI + Supabase Postgres + pgvector
- 배포: Vercel (프론트) + Railway/Render (백엔드)
- 결제: 토스페이먼츠 (국내) + Eximbay/PortOne (해외)

## D. Claude Code / Cursor로 이어서 작업할 때 권장 프롬프트

### D-1. 본 데모 코드를 발전시킬 때
```
다음 Streamlit 데모 코드를 기반으로 동일한 동작을 하는 React + FastAPI PWA로 마이그레이션해줘. 사관 페르소나 프롬프트와 사료 증거 카드 UI는 그대로 유지하고, 추가로 카카오맵 SDK + Geolocation API + Service Worker(오프라인)를 통합해줘. RAG는 ChromaDB가 아니라 Supabase pgvector로 구현해줘.
```

### D-2. RAG 평가셋 실행 자동화
```
core/rag.py 와 core/llm.py 를 활용해서, data/eval_set.json 의 50문항을 자동으로 실행하고 다음을 출력하는 평가 스크립트를 evaluate_rag.py 로 만들어줘:
- 카테고리별 통과율 (정답 가능 / 근접 오답 / 답변 불가 / 다국어 / 적대)
- 출처 정확도
- 평균 응답 시간
결과를 docs/eval_report.md 에 마크다운 표로 저장해줘.
```

### D-3. 다국어 결제·안전 문구 검수
```
프론트엔드의 모든 사용자 문구(결제·환불·안전·이동 안내)를 한·영·일·중 간체 4개 언어로 추출해서 docs/i18n_check.md 표로 정리해줘. 각 언어별 원어민 검수가 필요한 문구에 ⚠️ 마크를 붙여줘.
```

## E. 자주 빠지기 쉬운 함정

1. **API 키를 코드에 하드코딩 → 절대 금지**. 반드시 `.env`만 사용.
2. **샘플 사료의 `original_text`를 통째로 RAG 컨텍스트에 삽입 → 비용 폭증**. 요약만 사용하고 원문 링크 제공.
3. **사실 확인 배지를 LLM이 일관되게 출력하지 않으면 → `core/badge.py`의 폴백 로직 의존**. 시연 전 50회 정도 직접 호출해서 형식 안정성 확인 권장.
4. **다국어 답변 시 사료 인용이 빠질 수 있음 → `prompts.py`의 "응답 언어"와 "사료 인용" 규칙을 LLM에 강하게 강조**. 시연 전 영·일·중 답변에서 출처 카드 노출 확인.
5. **Streamlit 캐시(`@st.cache_data`) 사용 시 RAG 결과가 정적으로 고정될 위험 → 본 데모는 캐시 미사용**. 사료 추가 시에도 즉시 반영.
